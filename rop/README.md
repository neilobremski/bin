# Recursive Organic Processor (ROP)

## Executive Summary

The Recursive Organic Processor (ROP) is a digital organic system designed to solve the focus drift and context muddying inherent in large-scale LLM tasks. Instead of treating a prompt as a single monolithic execution, the ROP treats it as a parent objective that recursively spawns isolated child frames.

This mimics a biological call stack where work is localized, and only essential "genetic data" (state) is bubbled up.

## 1. Purpose and Vision

The ROP is intended to keep complex tasks coherent by decomposing them into smaller, isolated units of work. Each unit operates with only the context it needs, which reduces noise and helps preserve signal as the system scales.

## 2. Core Architecture: The "Inherit-Discover-Curate" Loop

The system operates on three primary layers of logic:

- **Inheritance (Descent):** A child frame inherits a filtered knowledge set from its parent. It has no access to global state, only what is necessary for its immediate micro-goal.
- **Execution (Leaf Nodes):** If a task is atomic, it is executed. The model then performs a self-discovery step to identify new facts, such as a variable definition or a character name.
- **Curation (Ascent):** The parent receives the output and the discoveries. The parent acts as a gatekeeper, deciding which discoveries are promoted to the branch's knowledge base and which are discarded as local noise.

## 3. Technical Guidelines and Constraints

| Constraint | Specification |
| --- | --- |
| Model | Qwen-3.5, optimized for line-based planning and IFBench performance |
| Telemetry | Telemetry on `STDERR` (`>&2`), data on `STDOUT`; this is critical for recursive variable capture |
| State Management | No global files. State is passed as an argument and stored in temporal JSON frames (`DDHHMMSS_RAND.json`) |
| Format | Line-based Markdown or pseudo-code. Avoid XML or JSON for planning to reduce token friction; use `jq` for structure |
| LLM Settings | `--think=false`, `--nowordwrap`. Reasoning should be externalized into the script logic, not the model's internal chain of thought |

## 4. Component Breakdown

- **The Membrane (Validator):** Every sub-task generated by a plan is run through a boolean gate to ensure it is not a hallucination or a truncated line before recursion begins.
- **The Immune System (Scorecard):** Leaf nodes use a retry loop. The model evaluates its own output against the task requirements and must return `YES` before the result is accepted.
- **The Temporal ID:** Frames are prefixed with the date format `+%d%H%M%S`. This creates a chronological audit trail in the [frames](./frames) directory.

## 5. Key Areas for Testing

- **State Drift:** Ensure that `CURRENT_LOCAL_KNOWLEDGE` is actually being updated and passed to siblings correctly.
- **Synthesis Quality:** Verify that the integration phase does not lose the granular details provided by the leaf nodes.
- **Parsing Robustness:** Improve the `sed` and `grep` logic used to separate `---RESULT---` from `---DISCOVERIES---` so it can handle cases where the LLM might output those strings unexpectedly.
- **Infinite Loop Prevention:** Test the `MAX_DEPTH` guard and implement a secondary breadth guard if the plan generates too many sub-tasks.
